# История тестирования пользователей

Моделирует пользователей, тесты, попытки и историю прохождений; подсказывает API-формы, UI списков и согласованность данных. Бэкенд — отдельный Django-репозиторий.

## Бэкенд Django (другой репозиторий)

- Не добавлять в этот проект Django-код, модели БД и миграции.
- Поля и URL эндпоинтов брать из согласованной спецификации или ответов реального API; при изменении контракта — синхронизация с бэкенд-разработчиком.
- Типичный JSON от Django REST framework: `snake_case`, пагинация (`count`, `next`, `results`) если так договорились.

## Модель данных (ориентир)

1. **User** — идентификатор, минимум PII в логах и типах для фронта.
2. **Test** (шаблон) — `testId`, название, версия схемы вопросов при необходимости.
3. **Attempt** — одно прохождение: `attemptId`, `userId`, `testId`, `startedAt`, `completedAt?`, `status`.
4. **Response / Step** — ответы внутри попытки с порядком или `questionId`; не смешивать попытки разных `testId` в одной записи без явной связи.

При изменении схемы теста хранить `schemaVersion` на попытке или снимок вопросов, чтобы история оставалась интерпретируемой.

## Потоки

- **Начало**: создать `Attempt` со статусом `in_progress`, зафиксировать `startedAt`.
- **Промежуточное сохранение** (опционально): PATCH шагов/idempotent по `attemptId` + `stepId`.
- **Завершение**: статус `completed`, `completedAt`; после этого не перезаписывать ответы без аудита.

## UI истории

- Сортировка по `startedAt` или `completedAt` убывание.
- Показывать название теста, дату, статус, длительность при наличии полей.
- Пустое состояние: «Пока нет пройденных тестов».

## Безопасность

- Загрузка истории только для текущего пользователя; идентификатор пользователя из сессии/токена на сервере, не из необоснованного query-параметра.
- Не кэшировать персональную историю в публичном CDN без явной политики.

## Чеклист перед мержем фичи истории

- [ ] Типы отражают nullable для незавершённых попыток
- [ ] Часовые пояса согласованы (хранение UTC)
- [ ] Нет утечки чужой истории при смене аккаунта (инвалидация кэша клиента)
- [ ] Поведение и коды ошибок API согласованы с Django или обработаны явно на фронте
